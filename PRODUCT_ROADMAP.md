# 产品落地路线

## 当前定位

这个工具的第一阶段不是做“面向所有学科的学术平台”，而是做你自己和相似用户真正会反复使用的投稿决策工具。

核心用户：

- 国际关系 / 国际政治方向研究生、博士生
- 同时面对中文核心 / SSCI 投稿压力
- 对期刊生态不完全熟悉
- 希望减少反复改格式和盲投试错

## 当前已经具备的能力

- 录入作者画像与稿件画像
- 自动识别主题、语言、模板化风险
- 中文核心 / SSCI 分别排序推荐
- 为每本期刊给出匹配理由和格式短板
- 本地保存稿件版本
- 本地保存某次投稿判断结果

## 最小可行产品的完成标准

满足以下 5 条，就可以算“可落地的自用 MVP”：

1. 能持续保存不同论文版本，而不是一次性分析
2. 能记录某次分析时的推荐结果，避免遗忘
3. 能对同一篇稿件切换不同投稿策略
4. 能对不同期刊输出明确的修改方向
5. 数据来源有基本可解释性

当前版本已完成 1-4 的基础形态。

## 接下来最值钱的迭代顺序

### Phase 1：你自己每天会用

目标：把它变成你的真实工作台。

优先做：

- 导入 docx 后自动抽取标题、摘要、关键词、篇幅
- 给每篇稿件维护“首投 / 备投 / 已拒 / 改投中”状态
- 给期刊增加更细的标签：审稿周期、是否常见博士独作、是否偏政策、是否偏理论
- 支持导出某本期刊的改稿清单

### Phase 2：从工具变成产品

目标：让同类用户也能直接上手。

优先做：

- 用户引导页和默认画像模板
- 可筛选的期刊库
- 期刊详情页
- 对不同目标期刊生成不同格式版本
- 更稳定的数据更新机制

### Phase 3：形成壁垒

目标：不是“模型推荐”，而是“有证据链的投稿判断”。

优先做：

- 区分官方规则与社区经验
- 给每条判断加置信度
- 沉淀真实投稿反馈
- 用历史投稿结果反向修正推荐逻辑

## 技术上建议怎么做

### 现在

- 静态前端 + 本地存储
- 适合快速试错、快速改产品逻辑

### 下一步

- 前端保留
- 增加轻量后端：用户、稿件、期刊、分析记录
- 独立数据层：期刊规则、投稿经验、结构化画像

### 再下一步

- 文稿解析服务
- 期刊规则引擎
- 推荐解释模块
- 反馈学习模块

## 你现在最该做的不是

- 一开始就接一键投稿
- 一开始就覆盖所有学科
- 一开始就做复杂账号体系
- 一开始就做大模型自动改全文

## 你现在最该做的是

- 把 10-20 本你最关心的期刊做深
- 用你自己的论文反复测试
- 把“判断为什么成立”说清楚
- 把“改到什么程度才适合投”说清楚

## 结论

这个产品是可以落地的。

但正确落地方式不是先做大，而是先把你自己的真实投稿流程吃透，再把它产品化。当前这套工具已经进入了这个路径：它不再只是概念，而是在往可持续使用的工作台演进。
